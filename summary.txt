# Inference Service Framework 기술 스펙 및 구조 명세서 (통합본)

이 문서는 Embodied AI 모델 평가 MLOps를 위한 **단일 서비스 추론 루프(Single-service inference loop) 프레임워크**의 통합 기술 스펙 정리본입니다. 첫 번째 정리본의 핵심 아키텍처 원칙 및 다방면의 논의 안건과, 두 번째 정리본의 심층적인 구조적/기능적 제약 사항을 모두 통합하여 구성했습니다.

---

## 1. 핵심 아키텍처 원칙 (Core Architectural Principles)

설정(Config)만으로 모델 런타임과 로봇 연동 방식을 유연하게 교체할 수 있도록 다음 세 가지 핵심 원칙을 따릅니다.

*   **관심사 분리 (Core vs Plugin):** 프레임워크 **코어는 실행(서비스 루프 관리, 라이프사이클 제어, 로깅, 설정 검증)만을 소유**합니다. 로봇의 조인트 제어나 모델의 텐서 처리와 같은 **구체적인 의미론적(Semantic) 동작은 모두 플러그인이 소유**합니다.
*   **완전한 모듈화 (Swappability):** 코어 코드의 수정 없이 YAML 설정 파일 조작만으로 입력/출력 어댑터, 매퍼, 모델 런타임, 프로세서 체인을 교체할 수 있습니다.
*   **의존성이 배제된 Fake-First 실행 체계 최우선:** ROS2, DDS, 하드웨어, GPU, 무거운 모델 체크포인트 없이도 로컬 환경에서 코어 아키텍처와 루프를 스모크 테스트(Smoke Test) 할 수 있는 가짜(Fake) 플러그인 파이프라인을 가장 먼저 구현하고 유지합니다.

---

## 2. 시스템 아키텍처 및 파이프라인 구조 (System Architecture & Pipeline)

프레임워크는 외부 소스로부터 데이터를 받아와 의미를 변환하고, 모델 추론을 거친 뒤 다시 로봇 명령으로 내보내는 단방향 순차 처리 파이프라인 구조를 가집니다.

### 2.1 전체 시스템 구조 (System View)

```mermaid
graph TD
    subgraph External[External Environment]
        Src((External Source<br/>ROS2 / Replay))
        Sink((External Sink<br/>DDS / ROS2))
    end

    subgraph Framework Core [Inference Service Loop]
        direction TB
        IA[Input Adapter] -->|Raw Observation| OM[Observation Mapper]
        OM -->|Canonical Observation| Pre[Preprocess Chain]
        Pre -->|Model Input| MR[Model Runtime]
        MR -->|Model Output| Post[Postprocess Chain]
        Post -->|Canonical Action| AM[Action Mapper]
        AM -->|Robot Command| OA[Output Adapter]
    end

    Src --> IA
    OA --> Sink

    classDef core fill:#f9f2f4,stroke:#d9534f,stroke-width:2px;
    classDef plugin fill:#dff0d8,stroke:#5cb85c,stroke-width:2px;
    class IA,OM,Pre,MR,Post,AM,OA plugin;
```

### 2.2 컴포넌트별 상세 역할 및 기능적 제한 (Component Roles & Limits)

모든 컴포넌트는 인터페이스 규약에 따라 엄격히 제한된 기능만 수행해야 합니다.

*   **Adapters (`InputAdapter`, `OutputAdapter`):** 외부 시스템과의 I/O 및 통신 라이프사이클 관리만 담당하며, **데이터의 의미를 파악하거나 변환해서는 안 됩니다**.
*   **Mappers (`ObservationMapper`, `ActionMapper`):** 로봇 종속적인 데이터(예: 29DOF 관절 배열 등)를 프레임워크 표준 데이터로 변환합니다. **특정 모델을 위한 이미지 정규화(Normalization)나 텐서 변환은 여기서 수행할 수 없습니다**.
*   **Processors (`Preprocess`, `Postprocess`):** 이미지 리사이즈, 정규화, 로봇 안전을 위한 리미트 클램핑(Clamp) 등 파이프라인 상의 단일 변환 및 검증을 체인 형태로 순차 수행합니다.
*   **Model Runtime:** 실제 모델 아티팩트 메모리 로드 및 추론만 전담합니다. **프레임워크 라이프사이클이나 로봇 통신 로직에 개입해서는 안 됩니다**.

## 3. 데이터 흐름 및 규격 (Data Flow & Canonical Model)

프레임워크는 로봇 데이터와 모델 데이터의 결합도를 낮추기 위해, 컴포넌트 간 경계마다 규격화된 의미론적 컨테이너(Canonical Data)를 사용합니다.

```mermaid
sequenceDiagram
    autonumber
    participant Ext as External Source/Sink
    participant Adapters as Adapters (I/O)
    participant Mappers as Mappers (Robot Specific)
    participant Procs as Processors (Core Chain)
    participant Runtime as Model Runtime

    Ext->>Adapters: Read
    Adapters->>Mappers: RawObservation (소스 특화 패킷)
    Mappers->>Procs: CanonicalObservation (표준 관절/카메라 포맷)
    Procs->>Runtime: ModelInput (PyTorch Tensor 등 모델 특화)
    Note right of Runtime: Inference 실행
    Runtime->>Procs: ModelOutput (Tensor 등 모델 특화)
    Procs->>Mappers: CanonicalAction (표준 목표 관절 위치 등)
    Mappers->>Adapters: RobotCommand (로봇 특화 제어 패킷)
    Adapters->>Ext: Publish
```

*   **Canonical Data Model 정책:** 모델/로봇 간 데이터 변환 시 프레임워크가 규정한 공통 필드(이미지, 조인트 등)를 사용합니다. 로봇이나 모델의 특수한 비표준 메타데이터는 확장성을 위해 `extras` 딕셔너리에 보관합니다.

## 4. 런타임 라이프사이클 및 로깅 (Runtime Lifecycle & Observability)

단순 제어가 아닌 '모델 평가'가 주 목적이므로, 명시적인 상태 머신과 추적 가능한 로깅을 제공합니다.

### 4.1 상태 머신 (State Machine)

리소스 낭비와 런타임 오류 방지를 위해, 설정 검증(`configure`)이 완전히 끝나야 리소스를 로드하도록 설계되었습니다.

```mermaid
stateDiagram-v2
    [*] --> CREATED: Init
    CREATED --> CONFIGURED: configure() (설정 및 플러그인 검증)
    CONFIGURED --> MODEL_LOADED: load_model() (메모리 할당)
    MODEL_LOADED --> WARMED_UP: warmup() (초기 예열)
    WARMED_UP --> RUNNING: run() (루프 실행)
    
    RUNNING --> PAUSED: pause() (명령 발행 중단)
    PAUSED --> RUNNING: resume()
    
    RUNNING --> ERROR: Unrecoverable Error
    RUNNING --> STOPPED: stop()
    PAUSED --> STOPPED: stop()
    ERROR --> STOPPED: stop() (강제 클린업 시도)
```

*   **Warmup (예열):** 정상 추론 시작 전, 유효한 샘플 Input을 통해 모델을 빈 상태로 1회 이상 실행합니다. 이때 생성된 결과는 로봇으로 발행되지 않고 버려집니다.
*   **Pause (일시정지):** 리소스는 로드된 상태로 유지하되, Output Adapter를 통한 로봇 명령 발행(Publish)만 즉각 차단합니다 (디버깅 안전망).

### 4.2 평가 궤적 기록 (Evaluation Traceability Logging)

매 반복(Iteration)마다 모델 런타임 종류, 추론 지연시간(Latency), 플러그인 에러 및 라이프사이클 이벤트가 JSONL 형태의 구조화된 로그로 저장됩니다. 원본 이미지나 거대한 텐서는 과부하를 막기 위해 로그에 직렬화(Serialize)하지 않습니다.

## 💡 기술 논의를 위한 제언 (Discussion Points)

아키텍처 및 엔지니어링 설계 단계에서 로봇, 모델, 인프라 유관 부서가 함께 협의해야 할 6가지 주요 안건입니다.

1.  **의존성 격리 정책 (Dependency Isolation)**
    *   *이슈:* ROS2, DDS, PyTorch, TensorRT 같은 무거운 의존성들이 프레임워크 코어에 섞이면 가벼운 테스트와 실행이 불가능해집니다.
    *   *논의점:* 의존성이 누수되지 않도록 플러그인 패키지를 어떻게 분리할 것인지, 선택적 플러그인(Optional Plugin)의 모듈 로딩 시점에서의 에러 처리 기준 수립.
2.  **`extras` 필드 통제 및 거버넌스 (Extras Policy)**
    *   *이슈:* Canonical 포맷에 맞지 않는 비표준 특수 데이터는 모두 `extras` 필드에 보관됩니다. 장기적으로 이 공간이 무분별한 "쓰레기통(dumping ground)"이 될 위험이 있습니다.
    *   *논의점:* 반복적으로 사용되는 `extras` 데이터를 어떤 시점과 기준으로 정식 Canonical 필드로 승격(Promotion)시킬 것인지 규칙 마련 필요.
3.  **에러 복구 및 예외 처리 정책 (Error & Resilience Policy)**
    *   *이슈:* 현재 MVP 명세는 입력 타임아웃(Input Timeout) 정도를 제외한 거의 모든 예외(플러그인 실패, 모델 추론 실패 등)를 복구 불가능한 `ERROR` 상태로 간주하고 즉시 루프를 멈춥니다.
    *   *논의점:* 로봇 제어 중 발생 가능한 통신 단절이나 일시적 예외를 '프레임워크 코어'에서 Retry 할지, 아니면 '플러그인 내부'의 책임으로 위임할지 경계 결정 필요.
4.  **루프 주파수 제어 보장성 (Loop Frequency & Real-time Constraints)**
    *   *이슈:* 제어 주기 설정(`loop_hz`)은 지원하지만 현재 설계상 하드 리얼타임(Hard Real-time) 타이밍을 보장하지는 않습니다.
    *   *논의점:* 모델 추론 지연(Latency)이 목표 제어 주파수를 초과할 때(Overrun), 데이터를 드롭(Drop)할 것인지, 혹은 버퍼링 우회 구조를 도입할 것인지 타이밍 전략 필요.
5.  **설정 파일(Config)의 동적 변경 허용 여부 (Dynamic Reconfiguration)**
    *   *이슈:* 현재는 모델 리소스 로드 이전(`configure` 단계)에만 설정 파일을 로드하여 검증하고 확정하도록 설계되었습니다.
    *   *논의점:* 로봇 실행 중(`RUNNING`)에 조인트 리미트 클램프 값 등의 특정 Processor 파라미터를 동적으로 변경할 수 있는 백도어를 열어둘지, 안전을 위해 프로세스 재시작만 허용할지 보안/안전 기준 수립.
6.  **Multi-stage 파이프라인 확장 대비 (Future Scalability)**
    *   *이슈:* 현재 범위는 단일 모델 루프(Single-stage)로 제한했으나, VLA 모델이 RL 에이전트를 호출하는 식의 병렬 또는 다단계 확장을 미래 목표로 명시하고 있습니다.
    *   *논의점:* 향후 Graph 기반 파이프라인으로 전환 시 현재의 순차적인 체인 구조(Processor Chain)에 기술 부채가 쌓이지 않도록 할 설계적 여유 방안 고려.
